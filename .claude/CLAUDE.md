# Obsidian Vault 運用ガイド

## ディレクトリ構成と運用方針

### 基本方針

- **MOC（Map of Contents）は使用しない**
  - 過去に検討した結果、ディレクトリ管理を複雑にしたくないため不採用

### ディレクトリ構成

#### `data/`
- **用途**: ナレッジベースの構築
- **運用方法**: Web Clipper を使用して AI が要約した内容を格納
- **目的**: 外部情報を体系的に蓄積する

#### `log/`
- **用途**: 学習セッションの記録
- **運用方法**: `data/` の内容を AI に質問してもらい、ユーザーの回答をまとめたドキュメントを保存
- **目的**: インタラクティブな学習プロセスを可視化する

#### `notes/`
- **内容**: 過去にインプットした情報をまとめ直したドキュメント
- **位置づけ**: vault 統合時に既存のメモから移行したもの
- **運用**: 参照用として保持（上記の運用フローとは異なる経緯で作成）

### Claude Code での作業時の注意点

- 新規ドキュメント作成時は、上記の運用フロー（`data/` または `log/`）に従うこと
- `data/` と `log/` の使い分けを意識すること
- `notes/` にある既存ドキュメントは参照のみで、積極的な編集は推奨しない（運用フローが異なるため）

### 双方向リンク運用ルール（追記案）

双方向リンクは 学習文脈を残すために使用する
- data/ は参照される前提で作成する
- log/ は必ず文章中でリンクし、リンク単体は置かない

リンクは「なぜ関連するか」が後から読んで分かる形で記述する

## 学習 × 開発の統合運用フロー

### 概要

「分からない」を起点に、学習と開発を一本で回し続ける運用フロー。

**全体の流れ**:
```
要約
 ↓
理解確認の質問
 ↓
分からないが出る
 ↓
最小実験（開発）
 ↓
言語化して保存
```

### 大原則（迷ったらここに戻る）

- ✔ 理解は あとからついてくる
- ✔ 分からないまま進んでいい
- ✔ 開発は「完成」じゃなく「確認」
- ❌ きれいにまとめない
- ❌ 全部理解しにいかない

### ① 学習フェーズ（インプット）

**やること**:
- CS50 / 記事 / 動画 / 本
- Web Clipper or AIで要約

**ガイドライン**:
- 要約は 粗くてOK
- 正確さより「話の芯」

### ② 理解確認フェーズ（NotebookLM）

**固定プロンプト（これ以外使わない）**:
```
この要約について、
私の理解を確認する質問を一問ずつ出してください。
```

**回答ルール**:
- 箇条書きOK
- 曖昧OK
- 分からなければ：「よく分からない / 今は追わない」
  👉 これが 次の開発の種

### ③ 開発フェーズ（最重要）

**開発の正体**: 質問に対する実験

**必須ルール（破らない）**:
- ⏱ 30〜60分
- 📄 ファイル1つ
- 🎯 確かめたいこと1つ
- 🧪 動けばOK

**開発テーマの決め方（テンプレ）**:
```
さっき分からなかった
「〇〇」を、分からないまま使ってみる
```

**開発中にやらないこと**:
- 設計を考える
- 正解コードを探す
- 最適化する

👉 「困るかどうか」だけを見る

### ④ 言語化フェーズ（1〜3分）

**Obsidian に残す型（固定）**:
```
【理解確認Q】
〇〇とは何か？

【実験で分かったこと】
・〇〇は知らなくても使えた
・△△は理解が必要だった

【今は追わない】
・〇〇（理由：使える）
```

※ 整えない
※ 書き直さない

### ⑤ 次に進む判断基準

✔ 動いた
✔ 1つ判断できた

→ 成功、次へ

### 分野を問わず使える理由

この型は常に：
- ❓ 質問を起点に
- 🧪 実験で確かめ
- ✍️ 自分の言葉で残す

だから：
- CS / Web / AI / キャリア / 設計 / 思考法

全部同じ運用で回せる

### よくある迷いへの即答

**Q. 理解が浅い気がする**
👉 正しい。あとで深くなる。

**Q. こんな小さな実験でいい？**
👉 いい。それが開発。

**Q. 特定の技術に寄りすぎない？**
👉 問題ない。抽象は後で効く。

### Claude Code での対応

Claude Codeは以下を理解してユーザーをサポートする：

1. **学習フェーズ**: 要約作業のサポート（必要な場合）
2. **理解確認フェーズ**: NotebookLMとの連携は想定しない（ユーザーが独自に実施）
3. **開発フェーズ（最重要）**:
   - 30〜60分で完結する最小実験を提案する
   - 「動けばOK」を基準に、過度な設計や最適化を避ける
   - ファイル1つ、確かめたいこと1つに絞る
   - ユーザーが「分からないまま使ってみる」姿勢を尊重する
4. **言語化フェーズ**:
   - 整形しすぎず、ユーザーの生の言葉を尊重する
   - 固定フォーマット（理解確認Q、実験で分かったこと、今は追わない）を意識する
5. **判断基準**:
   - 「動いた」「1つ判断できた」を成功と見なす
   - 完璧を求めず、次に進むことを優先する

**重要な姿勢**:
- ユーザーの「理解が浅い」「小さすぎる実験」への不安を否定せず、それが正しいプロセスであることを伝える
- 完成度より「試して確かめる」ことを優先する文化を尊重する
